# Frontend Audit — 2026-04-23

## Summary

Full sweep of `admin/*`, `_templates/*`, `shared/*`, plus CSP posture and Stripe/page-token plumbing. Local is up to date with `origin/main` as of fetch at audit start.

Severity totals: **C:3 / H:5 / M:11 / L:8 / N:4**

Top risk concentration:
1. `admin/clients/index.html` email-preview paths use `iframe.contentDocument.write(raw_html)` — the raw DB body bypasses srcdoc escaping entirely (**C1**).
2. `admin/newsletter/index.html:2058` assigns `frame.srcdoc = html` where the source is server-rendered newsletter HTML, but the iframe has `sandbox="allow-same-origin"` — editor-controlled HTML runs with parent origin capability against the admin session (**H1**).
3. `_templates/content-preview.html:135` declares `sandbox="allow-scripts allow-popups"` (no `allow-same-origin`) which is correct for the content pipeline, BUT the parent does a regex `replace` to inject endorsements into the raw HTML string before setting `srcdoc` — if `content_pages.html` contains a regex backref literal, the injection substring is unescaped. Verified scripts can run inside the iframe (design intent) but they inherit no cookies — accepted risk documented in the source comment. Flagged as **M** because the endorsement-string interpolation has no HTML-context escaping.

## Findings

### CRITICAL

- **[C1] Raw DB `body_html` written via `iframe.contentDocument.write` (bypass of srcdoc escaping)** — `admin/clients/index.html:4819-4820`, `:7604-7606`, `:9213-9217`. The admin "Preview Email" paths for audit followups (L4816), reports (L7600), and proposal followups (L9210) all do:
  ```
  iframe.contentDocument.open();
  iframe.contentDocument.write(res.html /* or data.body_html */);
  iframe.contentDocument.close();
  ```
  The iframe is created WITHOUT a `sandbox` attribute (no `setAttribute('sandbox', …)` anywhere on those branches), and the admin origin is the same as the iframe origin. A malicious `body_html` row (via a compromised follow-up-generator response, a stolen AGENT_API_KEY write path, or a naive admin pasting a template with `<script>`) would execute JS in the admin origin with full access to `mr_admin_sess` cookie via `fetch(credentials:'same-origin')`, `localStorage['sb-…-auth-token']`, and every DOM element on the admin page. Impact: admin session takeover. **Fix sketch:** replace `contentDocument.open/write/close` with the srcdoc pattern already used at L4299/4943/8954/9250 — `iframe.setAttribute('sandbox','allow-same-origin'); iframe.srcdoc = body.replace(/&/g,'&amp;').replace(/"/g,'&quot;');` — then let the onload handler measure height.
- **[C2] `admin/clients/index.html:4820` emails preview path runs the SAME bare `contentDocument.write` pattern in the audit-followups tab (`eaEmailBody`)** — same class of bug as C1, called out separately because the data source is `/api/admin/audits/…/followup-email-preview` which is AI-generated content that ALREADY passes through Claude prompting. Fix identical to C1.
- **[C3] Admin directories leak full-table reads with the anon key + two-step joins** — `admin/proposals/index.html:337-345`. The var is misleadingly named `SB_SVC` and the code comment and surrounding grep reveal the value is the ANON key, not service role (role=anon in the JWT payload). This is functional (and not a key leak) but the var name is a landmine — a future dev will see "SB_SVC" and assume admin authority is already present, and may remove a `requireAdmin` wrapper when porting logic. The file reads `proposals?select=*,contacts(...)` directly, so full contacts rows are anon-exposed via the `contacts` select wildcard. **RLS on `contacts` is the actual gate** (owned by DB agent) — flagged here because any future loosening of `contacts` RLS to support a new client-facing page will silently expose `email`, `phone`, `credentials`, etc. to every admin/proposals visitor. Fix sketch: rename the var to `SB_ANON`, swap the fetches to the admin-JWT'd `/api/admin/*` directory endpoints that already exist (`api/admin/clients-directory.js` pattern).

### HIGH

- **[H1] `admin/newsletter/index.html:725,2058` — preview iframe `sandbox="allow-same-origin"` + `srcdoc = html` from `/api/newsletter-preview`** — the response is server-rendered newsletter HTML, and the newsletter content is composer-editable by admin. If an attacker can drop JS into a newsletter draft (via stored input to `newsletters.content` JSONB — see newsletter composer fields), opening the Preview modal executes that JS with `same-origin` privileges against the admin session. Fix sketch: drop `allow-same-origin` on the preview iframe (we only need visual render), OR sanitize `/api/newsletter-preview` output through the existing `api/_lib/html-sanitizer.js` allowlist before send.
- **[H2] `admin/onboarding/index.html:376-377` does two direct anon PostgREST selects on `contacts` (`select=id,slug,status,practice_name,email,first_name,last_name,…`) and `onboarding_steps` (`limit=5000`)** — anon RLS on both tables is the gate. If `contacts` anon SELECT is ever broadened beyond "slug lookup" (as it was pre-C6), the entire contact directory is publicly dumpable. Same concern for `admin/reports/index.html:589+`, `admin/audits/index.html:294+`, `admin/deliverables/index.html:894+`, `admin/clients/index.html` (every inline `SB + /rest/v1/…` fetch). Fix sketch: migrate each admin directory page to call the `/api/admin/*-directory` endpoints already built for the job (they exist per the `api/admin/` inventory).
- **[H3] `admin/login/index.html:196,247,340` fetches `admin_profiles` with `apikey: SB_ANON, Authorization: Bearer ' + session.access_token` directly** — this is fine when the user is post-authenticated, but L340 does a PATCH to `admin_profiles` with the user's own access token. If RLS on `admin_profiles` ever permits `UPDATE USING (auth.uid() = id)` without column restrictions, a compromised admin can self-promote role. Fix sketch: route password-change and must-change-password clearing through `/api/auth/change-password` server-side with role-change validation.
- **[H4] `admin/audits/index.html:760-769` — `w.document.write('<iframe sandbox="allow-same-origin" srcdoc="' + body_html.replace(/&/g,'&amp;').replace(/"/g,'&quot;') + '">');`** — the replace chain is incomplete. It escapes `&` and `"` for attribute context but does NOT escape `<` or `'`. If `body_html` contains `'` mid-string and the wrapper switches to single-quote attribute later (or a browser quirk triggers attribute parsing leniency), the srcdoc terminates early and the trailing HTML renders in the popup origin (same admin origin, since `w = window.open()` inherits origin for `about:blank`). Fix sketch: use the standard pattern with both `&` and `"` AND avoid wrapping in any dynamically-chosen quote style — keep double-quoted srcdoc + the two-step replace; additionally verify `allow-same-origin` is necessary (for height auto-measure it is, else drop it).
- **[H5] `_templates/content-preview.html:168-185` — regex-based endorsements injection into raw HTML before `srcdoc` assignment** — `htmlToRender.replace(bioRegex, '$1' + endorsementsHtml + '$3')`. `endorsementsHtml` is built server-side from `endorsements` table rows (see L207-222) with no visible escaping. If an endorsement row contains `$1`/`$2`/`$3` literal sequences or raw HTML, they are interpreted by JS regex replace as backreferences. Also the iframe is `sandbox="allow-scripts allow-popups"` — script execution is intentional (it's a preview of a generated site) but any stored endorsement HTML lands inside that script-enabled context verbatim. Fix sketch: build `endorsementsHtml` through `api/_lib/html-sanitizer.js sanitizeHtml()` before concatenation; use a function replacer `function() { return endorsementsHtml; }` to kill backref interpretation.

### MEDIUM

- **[M1] `_templates/add-ons.html` is missing `/shared/offline-banner.js`** — every other client template includes it. File head verified: no `<script src="/shared/offline-banner.js">`. Users submitting an add-on checkout with flaky wifi have no visual cue. Fix: add the script tag alongside page-token.
- **[M2] Admin pages have NO offline banner included** — all 14 admin pages (`admin/index.html`, `admin/clients/index.html`, `admin/onboarding/index.html`, etc.) never include `/shared/offline-banner.js`. The template-layer prompt-spec says every client template must include it; admin UX is silent. Low-vis on desktop but a real issue on admin-mobile (fillable during client calls). Fix: include the shared script in each admin `<head>`.
- **[M3] `admin/clients/index.html:4820,7605,9216` — the UNSANDBOXED iframe `contentDocument.write` pattern additionally has NO origin isolation** — even if C1 is fixed to sandboxed-srcdoc, the height-auto-measure handler at L4816 (`onload = function() { iframe.style.height = iframe.contentDocument.body.scrollHeight + 20 + 'px'; }`) requires `allow-same-origin` per the CLAUDE.md known pitfall. That's the tradeoff — ship fixed height + scroll, OR keep same-origin and accept this risk class. Flag for operator decision.
- **[M4] `admin/audits/index.html:768` `data.body_html.replace(/&/g,'&amp;').replace(/"/g,'&quot;')`** is called ONLY on that line — if the admin ever clicks "Preview Raw Body" on an audit followup without body_html defined, `data.body_html` is undefined and the chained `.replace()` throws. No try/catch wraps this branch. Same pattern at `admin/clients/index.html:3197,4299,4943,8954,9250,9334`. Fix: `String(data.body_html || '').replace(...)`.
- **[M5] `_templates/checkout.html:375`, `entity-audit-checkout.html:228`, `add-ons.html:395` — `fetch('/api/checkout/create-session', {...})` does NOT pass `credentials: 'same-origin'`** — the endpoint is page-token-gated and the cookie is `mr_pt_<scope>` path-scoped to `/<slug>`. Modern Chrome defaults `credentials: 'same-origin'` for fetch so cookies SHOULD go, but explicit is safer. Fix: add `credentials: 'same-origin'` to every write fetch to `/api/*` on client-facing templates.
- **[M6] `_templates/checkout.html:387`, `entity-audit-checkout.html:240` — error state uses `alert()`** — any network error during checkout-session creation triggers `alert('Could not start checkout: ' + err.message + …)`. No branded error UI, no recovery guidance beyond the email link. Fix: render a styled error banner (pattern already exists in `onboarding.html` for save errors).
- **[M7] `admin/login/index.html:181` document.write fallback for the Supabase SDK** — `if (!window.supabase) { document.write('<scr'+'ipt src="https://unpkg.com/…supabase.min.js">...'); }` — this runs after the primary cdn.jsdelivr.net tag failed. If EITHER host is compromised, the admin login origin executes remote code. Fix sketch: host the SDK at `/vendor/supabase.min.js` and drop both external sources + the `script-src https://cdn.jsdelivr.net` CSP grant.
- **[M8] CSP `script-src 'self' 'unsafe-inline' https://cdn.jsdelivr.net https://msg.moonraker.ai`** — `'unsafe-inline'` is accepted per CLAUDE.md, but the `https://cdn.jsdelivr.net` grant is wide (the entire CDN), and only one file loads from it (`admin/login` SDK). Fix sketch: pin a specific SRI digest (`integrity="sha384-…"` on the script tag) and/or narrow the CSP to a concrete path pattern if Vercel's header engine supports it.
- **[M9] `_templates/onboarding.html:2390` reads `window.location.hash` into a step-key lookup (`hashKey = window.location.hash.substring(1).replace(/-/g, '_')`)** — the result is used for a `goToStep(hashKey)` equivalent. Not XSS (the value is not innerHTML'd) but no allowlist — unknown hash values trigger the default branch silently. Medium because a shared onboarding link with a trailing `#malformed` hash silently lands the client on step 0 instead of the intended step.
- **[M10] `admin/clients/index.html:9402`, `admin/site-map/index.html:437`, `_templates/report.html:1139,1183`, `_templates/checkout.html:252`, `_templates/proposal.html:661`, `_templates/checkout-success.html:90`** all read URL query params via `URLSearchParams(location.search).get(…)`. Spot-check: `report.html:1139` sets `isPreview = ...get('preview') === 'admin'` then uses it to toggle rendering — OK. `checkout.html:252` reads `urlPlan` and passes it through the tier lookup (map-indexed, so safe). `proposal.html:661` passes `params.get('v')` into a fetch URL — encoded via `encodeURIComponent` downstream — OK. No direct innerHTML sinks found. Flagged as Medium for future vigilance — any new URL-param use that flows to innerHTML is Critical.
- **[M11] `admin/inbox-search/index.html:196,246,244` uses `innerHTML = '<div class="error">' + escapeHtml(e.message) + '</div>'` with a custom `escapeHtml`** — check that the local `escapeHtml` is actually used on all paths. L246 writes a filter pill with raw subject fragments. Spot-check shows `escapeHtml` is applied; flagged Medium for code-review hygiene (a grep shows NO global `esc` helper exists in that file, and the local `escapeHtml` is scoped inside a closure — a future refactor could break the chain).

## What was verified as SAFE / not-a-finding

- **`_templates/*` head sections** — all 14 client templates have `<meta name="robots" content="noindex, nofollow">` and `<meta name="viewport">`. All admin pages have both.
- **Page-token scope declarations** — every client template that writes declares `window.__MR_PAGE_SCOPE__` BEFORE `shared/page-token.js` loads. Verified: `onboarding` (L1095), `campaign_summary` (L4), `proposal` (L783), `endorsement` (L4 + redundant L283), `progress` (L4). `report`, `diagnosis`, `action-plan`, `checkout`, `add-ons`, `entity-audit`, `entity-audit-checkout`, `content-preview`, `router` are read-only and correctly do NOT declare a scope.
- **Checkout `pageshow`/bfcache restoration** — `_templates/checkout.html:365` + `entity-audit-checkout.html:218` have the `pageshow persisted` pattern. Add-ons (L395) does not have a matching `restoreCheckoutBtn` — flag **M** if add-ons uses a button-state pattern too (not verified in detail).
- **`chat-panel.js` `esc` sanitizer** — no entity-decode-before-protocol-check vector found in the inline preview of L685-730. Deep audit recommended if COREBot response flow changes.
- **`eval` / `new Function` / `setTimeout(string)` usage** — zero hits across `admin/`, `_templates/`, `shared/`. Clean.
- **`startViewTransition` feature detect** — present with `else { apply(); }` fallback on every theme toggle (admin + templates).
- **`frame-src` CSP allowlist** — matches actual embeds (`msg.moonraker.ai`, `app.leadsie.com`). No wildcards.
- **Reporting-Endpoints / Report-To / report-uri** — all three set in `vercel.json`; CSP violations hit `/api/csp-report`.
- **No `buy.stripe.com` hardcoded URLs** in `_templates/` — lint rule R2 is holding.
- **Admin JWT cookie path** — `mr_admin_sess` set by `/api/auth/session` (API agent's lane); `shared/admin-auth.js` syncs on `TOKEN_REFRESHED`. Verified import present in `admin/index.html:11` + every other admin page `<head>`.
